# Aurum Agent Instructions

This file defines the operating rules for agents working on the Aurum Trading Platform repository.

## Source of truth

- Product and technical documentation live in the Slack Canvas: https://rvxsolutions.slack.com/docs/T0B4MRY2N8Y/F0B562EFLJU
- Project execution and task management live in Linear: https://linear.app/victormmelo/project/aurum-trading-plataforma-robo-btc-0ec902f0dca8

Treat the Slack Canvas as the central project document. It contains the product vision, MVP scope, trading strategy, architecture, data model, security model, MCP Server direction, dashboard expectations, success criteria, and roadmap.

Treat Linear as the source of execution state. Milestones, issue scope, priorities, status, and implementation sequencing should be managed there.

## Working rules

- Before implementing, read the relevant Slack Canvas sections and the matching Linear issue or milestone.
- Start implementation from a Linear issue whenever possible. If no issue exists, call out the need to create or update one before introducing meaningful scope.
- Keep the implementation aligned with the issue scope. If the code requires a scope change, update or propose an update to Linear and the Slack Canvas.
- When finishing work, report changed files, validations run, known limitations, and recommended next steps.
- Do not version real Binance credentials, API secrets, MCP tokens, production keys, or operational secrets.
- Preserve the current MVP scope unless the Slack Canvas and Linear explicitly say otherwise: Binance Spot Testnet, BTCUSDT, long-only, no leverage, no Mainnet execution.

## Documentation governance

Keep Slack, Linear, and the repository synchronized as the project grows.

Update the Slack Canvas when changing or deciding any of the following:

- Product scope or roadmap.
- Trading rules, risk rules, or promotion criteria.
- Architecture, services, data model, schemas, or endpoints.
- Security, permission, token, MCP, or audit behavior.
- Dashboard requirements, reporting behavior, or success criteria.

Update Linear when work changes status, priority, sequencing, ownership, milestone, or acceptance criteria.

## Frontend design direction

The frontend should use the design generated by:

```bash
npx getdesign@latest add mastercard
```

Before changing UI, check whether the generated design components, tokens, or patterns already exist in the project. Keep dashboards and operational screens aligned with that design system instead of creating a parallel visual language.

If the design command has not been run yet, create or update a Linear issue before introducing the design foundation. Do not silently add a different design system.

## Engineering guardrails

- Prefer small, issue-scoped changes over broad refactors.
- Keep documentation and implementation consistent with the Slack Canvas.
- Add or update tests when behavior changes.
- Run the narrowest meaningful validation for the change and record the result.
- For frontend work, preserve an operational dashboard experience rather than replacing it with a marketing landing page.
